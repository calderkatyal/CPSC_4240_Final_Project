\begin{abstract}
FlashAttention is the standard exact attention kernel used in modern large language models because it substantially reduces high-bandwidth-memory traffic while preserving exact attention outputs. In this project, we implement a compact CUDA C++ attention benchmark suite centered on the most important broad ideas from FlashAttention-1: shared-memory tiling, online softmax, fused score/softmax/value computation, mixed precision, and tensor-core score tiles. We then add one carefully scoped extension motivated by later versions: a split-KV sequence-parallel path that preserves exact attention while changing how the KV range is partitioned. Our goal is not to reproduce the full production codebase; rather, it is to evaluate how much of FlashAttention's forward-pass advantage can be captured by a compact kernel that preserves the main FA1 design choices, and then to measure how much additional benefit is provided by a later-version split-KV modification once that FA1 core is already in place. The resulting benchmark suite compares a shared PyTorch eager-attention baseline, a simplified FlashAttention-1 kernel, a split-KV forward extension, ablations of the most important FA1 design decisions, and two official baselines: FlashAttention-1 and FlashAttention-2.
\end{abstract}

\section{Introduction and Motivation}
Self-attention is a central primitive in transformer models, but the naive implementation is expensive in both computation and memory traffic. Given queries $Q \in \mathbb{R}^{N \times d}$, keys $K \in \mathbb{R}^{N \times d}$, and values $V \in \mathbb{R}^{N \times d}$, standard attention computes
\begin{equation}
\operatorname{Attn}(Q,K,V) = \operatorname{softmax}\left(\frac{QK^\top}{\sqrt{d}}\right)V.
\end{equation}
The bottleneck is not only the $O(N^2)$ arithmetic from the score matrix $QK^\top$, but also the cost of writing and rereading the full $N \times N$ matrix from high-bandwidth memory. FlashAttention addresses this by reorganizing exact attention so that score computation, normalization, and value accumulation are fused over tiles, avoiding explicit materialization of the full attention matrix \citep{dao2022flashattention}.

Our project goal is narrower than reimplementing the entire official repository. We instead aim to answer three concrete questions.
\begin{enumerate}
    \item Which FlashAttention-1 ideas are most necessary for a meaningful modern GPU benchmark?
    \item Once a simplified FlashAttention-1 kernel is implemented, does an exact split-KV extension still help?
    \item How large is the gap between a compact educational CUDA implementation and the production \texttt{flash\_attn} module?
\end{enumerate}
To answer these questions, we implement a GPU-only benchmark suite with a shared PyTorch eager-attention baseline, a simplified FlashAttention-1 core, one exact split-KV extension, and four ablations that remove key FA1 ingredients. The benchmark suite is designed to compare these project kernels against both official FlashAttention-1 and official FlashAttention-2 on the same CUDA machine.

\section{Background}
\subsection{Attention and the memory bottleneck}
For a single head, standard attention computes three conceptual stages: score formation, row-wise normalization, and value mixing. In matrix form,
\begin{align}
S &= \frac{QK^\top}{\sqrt{d}}, \\
P &= \operatorname{softmax}(S), \\
O &= PV.
\end{align}
The mathematical definition is simple, but the straightforward implementation is expensive on GPUs because it repeatedly reads and writes the full $N \times N$ score matrix. This cost becomes especially severe at long sequence length, where the score matrix dominates both memory traffic and intermediate storage. FlashAttention's central insight is that exact attention can be reorganized around the GPU memory hierarchy: instead of storing the full score matrix, it computes attention tile-by-tile while carrying only the sufficient per-row normalization statistics needed to preserve exactness \citep{dao2022flashattention}.

\subsection{Why online softmax gives exact attention}
The key algebraic trick behind FlashAttention-1 is that softmax normalization can be updated incrementally across score tiles without approximation. Suppose a row has already processed some subset of keys and currently stores a running maximum $m$ and a running denominator $\ell = \sum_j e^{s_j - m}$. When a new tile of scores arrives, let $\hat{m}$ be the maximum score inside that tile and define the new row maximum as $m' = \max(m, \hat{m})$. Then the updated denominator can be written exactly as
\begin{equation}
\ell' = e^{m - m'} \ell + \sum_{j \in \text{new tile}} e^{s_j - m'}.
\end{equation}
The same rescaling factor $e^{m - m'}$ can be applied to the running output accumulator before adding the new tile's contribution. This recurrence is what lets FlashAttention remain exact while discarding the old score tile once its contribution has been folded into the running state \citep{dao2022flashattention}. In other words, the algorithm reduces the memory footprint by changing the order of computation, not by changing the mathematical attention rule.

\subsection{Evolution from FlashAttention-1 to FlashAttention-4}
The FlashAttention family has evolved in a way that is useful for framing our project. FA1 contributes the core IO-aware exact-attention reformulation. FA2 improves forward performance through better work partitioning and reduced overhead beyond raw matrix multiplication \citep{dao2023flashattention2}. FA3 is much more architecture-specific: its key contributions are asynchronous execution, warp specialization, and deeper co-design with Hopper hardware \citep{shah2024flashattention3}. FA4 pushes this hardware/software co-design further for asymmetric scaling on newer systems, emphasizing redesigned fully asynchronous MMA pipelines, larger tiles, and reduced non-matmul bottlenecks such as exponential evaluation \citep{zadouri2026flashattention4}. These later versions are important context for understanding the trajectory of the family, even though our implementation target remains a compact FA1-centered kernel plus a single split-KV extension.

This scoping decision matches our experimental goal. We are not training models, so we prioritize the features that most strongly affect forward-pass performance:
\begin{itemize}
    \item no explicit score-matrix materialization,
    \item shared-memory tiling,
    \item online softmax,
    \item fused exact attention,
    \item mixed precision, and
    \item tensor-core-friendly score computation.
\end{itemize}
These are the parts of FlashAttention that are both conceptually central and broadly relevant on modern NVIDIA GPUs.

\section{Methods}
\subsection{PyTorch eager-attention baseline}
Our shared baseline matches the eager PyTorch attention path used in the official FlashAttention benchmark script. It stacks $Q,K,V$ into a packed tensor, forms the full score matrix with a batched matrix multiply, applies the causal mask and row-wise softmax, and then multiplies the resulting attention probabilities by $V$. The implementation uses FP16 inputs so that it is compared against the project kernels under similar data-type assumptions, but it still materializes the full score matrix. This baseline therefore preserves exact attention math while exposing the major memory-system weakness that FlashAttention is designed to remove, and it provides one common denominator for both the project kernels and the official FlashAttention kernels.

\subsection{Simplified FlashAttention-1 kernel}
The main project kernel is our simplified FlashAttention-1 implementation. Each thread block owns a $64$-row query block composed of four warp-local $16$-row query tiles. For each outer iteration, the kernel stages the current query block together with a $64$-key / $64$-value block in shared memory, computes the score subtiles with WMMA tensor-core fragments, and maintains per-row running softmax statistics across KV blocks. Let $m_i$ and $\ell_i$ denote the running row maximum and running normalization term for query row $i$. For a new KV block producing scores $s_{ij}$, the online recurrence is
\begin{align}
m_i' &= \max\left(m_i, \max_j s_{ij}\right), \\
\ell_i' &= e^{m_i - m_i'} \ell_i + \sum_j e^{s_{ij} - m_i'}, \\
o_i' &= e^{m_i - m_i'} o_i + \sum_j e^{s_{ij} - m_i'} V_j .
\end{align}
After the final KV block, each output row is normalized by $\ell_i'$. This is the main algorithmic reason FlashAttention can remain exact while avoiding the full $N \times N$ attention matrix \citep{dao2022flashattention}.

Our kernel is intentionally smaller than the official implementation, but it preserves the core forward-pass structure:
\begin{itemize}
    \item a query block and the current key/value block are staged in shared memory;
    \item the score computation is built from WMMA tensor-core subtiles inside the wider staged KV block;
    \item softmax normalization and value accumulation are fused into the same kernel; and
    \item only per-row running state and the final output are kept live between KV blocks, without materializing the full score matrix.
\end{itemize}

\subsection{Split-KV extension}
The most important portable later-version idea we add is a split-KV forward path. Rather than assigning the full KV range for one query block to a single block, we partition the KV blocks across multiple blocks. Each block computes partial statistics for a subset of the KV blocks assigned to the same query block, producing partial maxima $m_i^{(r)}$, partial normalizers $\ell_i^{(r)}$, and partial output accumulators $o_i^{(r)}$ for split $r$. These are then combined exactly with
\begin{align}
m_i &= \max_r m_i^{(r)}, \\
\ell_i &= \sum_r \ell_i^{(r)} e^{m_i^{(r)} - m_i}, \\
o_i &= \frac{1}{\ell_i} \sum_r o_i^{(r)} e^{m_i^{(r)} - m_i}.
\end{align}
This extension is motivated by the upstream split-KV path and its \texttt{num\_splits} heuristic in the official implementation. It is not the entire FlashAttention-2 redesign. Native FA2's main gains come from broader work-partitioning changes and reduced non-matmul overhead, whereas our project extension isolates only the exact split/combine mechanism \citep{dao2023flashattention2}.

\subsection{Ablations}
We include four ablation kernels so that the report can answer which FA1 ingredients matter most.
\begin{itemize}
    \item \textbf{No tensor cores.} This kernel keeps the same staged query/KV tiles, online-softmax recurrence, and fused forward pass, but replaces the WMMA score/output updates with scalar per-thread math.
    \item \textbf{No vectorized loads.} This kernel keeps the same FA1 core algorithm, but disables the 16-byte Q/K/V tile loads used by the main path.
    \item \textbf{No online softmax.} This kernel keeps the same block structure, staged query/KV blocks, and tensor-core score subtiles, but replaces the online recurrence with a two-pass computation.
    \item \textbf{No SRAM tiling.} This kernel keeps the same outer block structure and exact online-softmax logic, but loads key/value rows directly from global memory instead of staging K/V blocks in shared memory. In our compact codebase this is the broadest structural ablation, since removing the staged K/V tile also removes the convenient tensor-core score path.
\end{itemize}
The ablations are important because they let us isolate whether the main gain comes from tensor-core math, vectorized tile movement, the online recurrence, or the memory-hierarchy reorganization. The first two ablations still preserve the main IO-aware fused structure and should often remain faster than eager PyTorch, while the latter two intentionally remove the dominant memory-saving ingredients and can legitimately fall below the eager baseline.

\subsection{Why we use a split-KV extension}
The later-version idea we chose to isolate is split-KV sequence parallelism. This choice is deliberate: it is a real exact-attention mechanism from the FlashAttention codebase, it is important enough to matter in low-occupancy settings, and it can be isolated cleanly in a student implementation. In contrast, many of the larger FlashAttention-2 improvements are more entangled with deeper kernel-engineering changes. Split-KV therefore gives us a clean extension to study without pretending that it is a full miniature of native FA2.

\subsection{Implementation details relevant to the GPU study}
Two implementation choices are especially important for interpreting the benchmark. First, all project kernels use FP16 inputs and FP32 accumulation. This keeps the benchmark aligned with the mixed-precision regime used by modern FlashAttention implementations while avoiding the additional complexity of BF16 support in our student codebase. Second, the head dimension is fixed to $d=64$ for the main benchmark sweep. This is small enough to be common in practice and is one of the common compile-time specializations we keep in the simplified code path, namely $\{32, 64, 128\}$.

The simplified FA1 kernel and the split-KV extension also share most of the same surrounding choices: both perform exact causal attention, both keep the output accumulation in FP32, and both use the same correctness reference in the benchmark binary. This is important because it means the FA1-versus-split-KV comparison is not confounded by datatype changes, approximation, or a different masking convention. The intended interpretation is therefore narrow and clean: if split-KV helps, it should help because of additional sequence-level parallelism and altered work partitioning in low-occupancy regimes, not because the kernels are doing different mathematical tasks.

To stay close to the official FlashAttention-1 source without making the codebase explode in size, the main FA1 path also adopts several low-level choices from upstream: compile-time dispatch for supported head dimensions, a native-like larger KV tile for the $d=64$ benchmark regime, 16-byte vectorized tile loads for $Q/K/V$, a shared K/V SRAM buffer with V fragments held in registers, and the $\exp_2$ / $\log_2 e$ softmax scaling path used by the native kernel. These are all symmetry-preserving implementation choices rather than benchmark hacks: they apply equally across sequence lengths and do not change the exact attention computation.

\subsection{Kernel layout}
Our CUDA implementation uses $16 \times 16$ WMMA fragments, but the full thread-block work unit is larger: each block processes a $64$-row query block and iterates over the sequence in $128$-key / $128$-value blocks. The query block and the current KV working set are staged on chip, with the value fragments loaded into registers before the shared K/V buffer is reused for the key tile. Each active query row maintains its running maximum, running normalization constant, and running output vector in FP32 across KV-block iterations. Once all KV blocks have been processed, the output accumulator is normalized and written back to global memory.

Figure~\ref{fig:hbm_sram_map} makes the memory-hierarchy mapping explicit: HBM stores the full $Q$, $K$, $V$, and $O$ tensors, while SRAM and other on-chip state hold only the current query tile, the current K/V tile, and the row-wise running state needed for exact online softmax. The full $N \times N$ score matrix is never materialized.

The split-KV kernel keeps the same intra-block computation but changes how the KV range is traversed. Instead of assigning the full KV range to one block, it partitions the KV blocks into splits. Each split produces partial $m$, $\ell$, and $o$ values, and a second kernel combines them exactly. This split/combine structure is the main implementation difference between our FA1 core and the extension path.

\section{Experimental Setup}
We study dense causal self-attention in a controlled forward-only setting. All methods are evaluated on the same tensor shapes and masking convention so that differences in runtime reflect kernel organization rather than changes in the mathematical task. The project kernels and the official baselines are all run in mixed precision with FP16 inputs. Output accumulation and the reference comparison are performed in FP32, which matches the numerically stable regime used throughout the FlashAttention literature \citep{dao2022flashattention,dao2023flashattention2}.

We report runtime, peak GPU memory, and maximum absolute error relative to an FP32 reference.

The evaluation reports four classes of comparisons.
\begin{enumerate}
    \item \textbf{PyTorch baseline vs.\ simplified FA1.} This isolates the main FlashAttention-1 design choices inside the project CUDA kernels.
    \item \textbf{Simplified FA1 vs.\ split-KV extension.} This isolates split-KV sequence parallelism.
    \item \textbf{Project kernels vs.\ official FlashAttention-1 and FlashAttention-2.} This measures the gap between the educational implementation and the production kernels from both generations.
    \item \textbf{FA1 ablations.} These isolate which of the core FA1 ingredients are most important in practice.
\end{enumerate}

The official baselines serve two distinct roles in this study. Official FlashAttention-1 anchors the comparison to the original IO-aware reformulation, while official FlashAttention-2 reflects the more mature forward kernel family that is widely used on modern NVIDIA GPUs.

\section{Discussion}
The evaluation is organized so that each comparison answers a specific systems question. The most important project-side comparison is the shared PyTorch baseline versus simplified FA1, since that isolates the practical effect of IO-aware tiling, online softmax, and score-matrix elimination in our compact kernel. The FA1-versus-split-KV comparison is intentionally narrower: it asks whether exact split-KV sequence parallelism with a combine step contributes additional benefit once the FA1 core is already in place. For the merged speedup summaries, all methods are normalized against the same PyTorch attention baseline so that the official kernels and the project kernels share one common reference point.

\section{Limitations}
This project still has several deliberate limitations, all of which should be acknowledged in the final presentation of results.
\begin{itemize}
    \item The code is forward-only and does not include training kernels.
    \item The kernels currently specialize the common head dimensions $\{32, 64, 128\}$ rather than the full set of multiples of $16$.
    \item The implementation uses FP16 inputs and FP32 accumulation, but does not yet support BF16.
    \item The split-KV extension is intentionally narrow and should not be presented as a full reproduction of FA2.
    \item The comparison against the official FA1 and FA2 packages is numerically meaningful and practically useful, but it is not an apples-to-apples feature comparison.
\end{itemize}
These limitations are acceptable for our stated project goal because that goal is to identify and benchmark the most important ideas behind FlashAttention's forward-pass behavior, not to reproduce the entire production stack.

\section{Conclusion}
This project studies how much of FlashAttention's practical forward-pass advantage can be captured in a compact CUDA C++ implementation. The main kernel preserves the central FlashAttention-1 design decisions: shared-memory tiling, online softmax, fusion, mixed precision, and tensor-core score computation. The additional split-KV path then tests whether sequence-parallel work partitioning offers a further gain once the FA1 core is already in place. Taken together, these kernels provide a focused benchmark suite for understanding which algorithmic and hardware-conscious ideas matter most for exact attention performance on modern GPUs.
